共轭生命周期​

缓存共轭对象的引用​

当你想缓存一个共轭对象的引用时，如果你明确知道它是黑色，那么你可以直接缓存它本身； 如果你明确知道它是红色，那么你应该缓存它的拷贝/克隆； 如果你不知道它的颜色，可以使用这种写法：

Vector vec = ...;
Vector cache = vec.IsBlack ? vec : vec.Clone();

如果你更关注代码的可读性，也可以无差别地缓存一份拷贝：

Vector vec = ...;
Vector cache = vec.Clone();

使用 UObject 共轭​

逻辑上，UObject 共轭的行为和 TWeakObjectPtr 完全相同：它可以查询对象的有效性，但不会影响垃圾回收。 和 TWeakObjectPtr 类似，在使用之前，你需要确保 IExplicitLifecycle.IsExpired 属性为 false。 需要注意的是，每当你从 await 语句中返回，你都应该重新判断，因为 await 语句可能是异步的。 当然，这意味着你也需要留意其他异步情况，如闭包中的 upvalue。

UObject 共轭对象不会影响原对象的回收，如果你想在托管侧添加对 UObject 的引用，可以使用以下写法：

UnrealObject obj = ...;
StrongObjectPtr<UnrealObject> strongObj = new(obj);

上述代码构造了一个 StrongObjectPtr 对象，并指向 obj。 它的效果和你在 C++ 中使用 TStrongObjectPtr 是相同的。 因此，在 strongObj 被销毁或指向其他对象之前，obj 都不会被回收。

上面提到，在托管侧，普通的 UnrealObject 引用和 WeakObjectPtr 拥有等价的功能。 并且，在查询有效性时，普通的 UnrealObject 引用比 WeakObjectPtr 快得多（不在一个数量级）。 虽然 Z# 导出了 TWeakObjectPtr 的共轭类型，但除了以下两种情况，你一般不会需要它：

  1. 调用引擎原生 API 或重写引擎原生的蓝图事件函数。
  2. 声明一个 WeakObjectPtr 类型的 UProperty，目的是在不影响对象回收的情况下支持 UE 反射。

对虚幻字符串和容器使用 LINQ​

虚幻字符串的共轭类型实现了 IEnumerable<char> 接口，虚幻容器的共轭类型实现了 IEnumerable<T> 接口。 这意味着你可以对它们使用 LINQ。 但是，你在对这些类型使用 LINQ 时必须格外小心，因为 LINQ 中的很多操作都涉及 惰性求值。 惰性求值在不可变系统中是极其强力的武器，但不幸的是，这是一个充满副作用的系统，集合随时有可能被改变，非托管对象随时有可能被销毁。 如果你在使用 LINQ 时没有正确处理对象生命周期，就可能产生未定义行为。

粗略地讲，对于所有返回抽象结果 IEnumerable<T> 的操作，我们建议立即进行消费，而不要对返回值进行缓存，因为这类操作大多数都使用了惰性求值，即等到真正遍历的时候才生成结果； 而对于返回具体结果的操作，如 int32 Count()、bool Any()、T[] ToArray() 等，则没有生命周期的问题，因为这类操作是强制求值，即在返回之前就必须遍历整个输入源。

当然，这只是给不熟悉 LINQ 的开发者的保守建议。如果你知道自己在做什么，就不需要遵循这些建议。

显式释放黑色共轭​

共轭类型都实现了 IDisposable 接口。 如果你在作用域中创建了一个黑色共轭对象，并且能确定它不会逃逸出该作用域，你可以使用以下写法来将它的有效期与作用域绑定，这样做可以略微减轻 GC 压力：

{
    using Vector vec = new(1, 2, 3);
    ...
    // vec will be disposed just before this scope exits.
}
注意

你不能显式释放一个红色共轭对象，因为你没有它的所有权。如果你这样做，将会抛出 InvalidOperationException。

错误处理​

仅将断言用于检查内部错误​

你应该只用断言检查程序的内部错误。而对于外部错误，如用户使用错误、不可控的错误，你应该抛异常。

断言和异常的区别在于是否可以被外部处理以及应不应该发生。 内部错误是你自己写的 bug，这意味着它们根本不应该发生。 这也是为什么发布版本中的断言会被忽略。 你应该从代码层面修复内部错误，而不是在运行时尝试恢复这类错误； 你的用户更不应该处理你的内部错误，他甚至不需要知道你发生了内部错误。 因此，你应该用断言来尽早暴露内部错误，而不是用异常把它们抛给用户，这没有意义，用户不知道怎么解决你的内部问题。 这也是为什么所有断言都伴随一条日志——断言是给人看的，而不是给计算机看的。

有时，为了保证开发体验，你可能需要在开发版本的运行时修复内部错误。 对于可以恢复的内部错误，推荐使用 ensure 系列的断言，这可以提高开发工具本身的健壮性； 对于无法恢复的内部错误，推荐使用 check 或 verify 系列的断言（取决于断言的内容有无副作用），这可以帮助你尽早发现严重问题。

不要捕获 FatalException​

当你的代码发生了不可恢复的内部错误时，你应该抛出 FatalException，使用断言会自动帮你做这件事。 由于这个错误是不可恢复的，你不应该去捕获它，而是应该让它流转到最外层并终止程序的运行。 FatalException 利用异常传播的特性，可以在中间层不知情的前提下将一些具体的错误信息带到最外层，让程序在崩溃时可以留下一些有用的信息帮助你排查问题。

不要在跨边界的代码块中抛异常​

在跨边界的代码块（这里特指从非托管侧跨到托管侧）中抛异常会导致异常最终流转到 CLR 内部导致进程终止，以这种方式终止进程时不会留下任何信息。 因此，如果你的跨边界代码块可能抛出异常，你应该使用以下写法：

try
{
    ...
}
catch (Exception ex)
{
    UnhandledExceptionHelper.Guard(ex);
}

这种写法会将普通异常转换成 Error 日志，并让 FatalException 以合理的方式终止进程。

使用 notnull 语义而不是 NullReferenceException​

nullable 特性可以极大地提高代码的可读性。 如果你的接口无法处理空值，即面对一个空值时只能抛出异常，那么你应该直接在签名里接收一个 notnull 的引用，而不是手写校验逻辑。 Z# 项目默认开启 nullable 特性，你可以手动关闭，但除非有特殊原因，否则不推荐这么做。

其他​

不要滥用 unsafe​

unsafe 特性允许你在托管代码中直接操作内存地址。 这个特性在 .NET 原生开发中是默认关闭的，但由于 Z# 中存在大量互操作代码，我们不得不默认开启 unsafe。 虽然 unsafe 很有用，但对于一个脚本引擎而言，安全性大于天。 如果程序从脚本中崩溃，所有 unsafe 代码都将成为疑点，这意味着 unsafe 代码越多，排查问题的难度就越大。 实际上，脚本根本就不应该能引发崩溃，因为它不仅是为了提升开发效率，也是为了降低使用门槛。

通常情况下，你不应该为了一些无关紧要的原因使用 unsafe，如微小的性能提升。 如果你必须使用 unsafe，推荐使用 unsafe 代码块，而不是将整个方法甚至整个类型标记为 unsafe。 将 unsafe 代码与安全的代码隔离开也是一个好习惯，因此，你也可以选择将 unsafe 代码封装到一个单独的 unsafe 方法中。

不要滥用 dynamic​

dynamic 特性允许你以动态类型的方式编写托管代码，即变量的类型与运行时值绑定，而不是与编译期签名绑定，也可以使用没有显式声明的类型。 编译器不会对 dynamic 代码进行类型检查，而是会像动态语言一样将检查延迟到运行时。 这个特性在处理一些重复性强的琐事时很有用，如大量的 BO <-> DTO 转换。 但滥用这个特性会失去 .NET 强大的静态特性，在一些高度被其他模块依赖的代码中使用甚至会导致传染。 因此，我们强烈建议你只在实现细节中使用 dynamic，永远不要在接口中使用。
